## 三、禁止事项清单

### 严格禁止 🚫

| 禁止项                  | 正确做法                   |
| ----------------------- | -------------------------- |
| 禁止使用静态JSON模拟API | 必须实现真实Express路由    |
| 禁止前端硬编码数据      | 必须从API获取              |
| 禁止跳过数据库操作      | 必须真实读写MongoDB        |
| 禁止忽略错误处理        | 必须try-catch+中文错误消息 |
| 禁止跳过权限验证        | 必须使用auth中间件         |
| 禁止测试用例造假        | 必须实际调用API验证        |

### 代码质量底线 🔴

| 问题类型    | 严重性  | 处理方式           |
| ----------- | ------- | ------------------ |
| SQL注入风险 | 🔴 致命 | 立即修复，禁止合并 |
| 无认证接口  | 🔴 致命 | 立即修复，禁止合并 |
| 无try-catch | 🔴 致命 | 立即修复，禁止合并 |
| 内存泄漏    | 🟠 严重 | 24小时内修复       |
| 无分页查询  | 🟠 严重 | 立即修复           |

## 四、开发执行命令

### 每个功能开发流程

```bash
# 1. 创建功能分支
git checkout -b feature/功能名称

# 2. 开发后端API
# - 在 routes/ 创建路由
# - 在 controllers/ 创建控制器
# - 在 models/ 创建/更新模型

# 3. 开发前端页面
# - 在 public/ 创建HTML
# - 在 public/js/ 创建JS

# 4. 本地测试
npm run dev  # 启动开发服务器
npm run lint # 代码检查
npm test     # 运行测试

# 5. 审查通过后合并
git add .
git commit -m "feat: 功能描述"
git checkout main
git merge feature/功能名称
```

## 五、验收测试用例

### 每个功能必须通过的测试

#### 用户认证测试

```bash
# 测试注册
curl -X POST http://localhost:3000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"test123","email":"test@example.com"}'

# 预期：返回201 + 用户信息

# 测试登录
curl -X POST http://localhost:3000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"test123"}'

# 预期：返回200 + JWT token

# 测试受保护接口
curl -X GET http://localhost:3000/api/users \
  -H "Authorization: Bearer <token>"

# 预期：返回用户列表
```

#### 知识库测试

```bash
# 测试创建知识
curl -X POST http://localhost:3000/api/knowledge \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"title":"测试知识","category":"basic","content":"内容","keywords":["测试"]}'

# 预期：返回201 + 知识ID

# 测试文件上传
curl -X POST http://localhost:3000/api/knowledge-management/upload \
  -H "Authorization: Bearer <token>" \
  -F "file=@test.pdf" \
  -F "title=测试文档" \
  -F "category=basic"

# 预期：文件保存到uploads/目录，返回文件信息
```
